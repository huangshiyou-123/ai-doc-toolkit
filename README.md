# AI 文档处理 — 产品实验

> 一个产品思维展示项目：从用户痛点出发，完整走完产品设计→功能定义→原型→上线的全流程。

---

## 这个项目是什么

这不是一个复杂的技术产品，而是一个**产品实验**——我用它验证两件事：

1. AI 文档处理的真正瓶颈不是模型能力，而是缺少固定输出框架约束 AI 的行为
2. 我能不能从0到1走完产品全流程：发现问题 → 定义需求 → 设计方案 → 落地能用

**这个项目做了什么：** 四个预设 Prompt 模板（文档摘要、需求整理、会议纪要、智能识别），配合浏览器端和命令行两种交互方式，解决"每次用 AI 处理文档都要重新写提示词、输出格式不稳定"的问题。

---

## 产品决策记录

做这个项目的过程中，有几个设计决策我自己踩过坑、纠结过，记录在这里：

### 1. 为什么是 Prompt 模板而不是更复杂的技术方案？

当时考虑过两种方案：接入 Dify 工作流引擎，或者直接用预设 Prompt + 固定输出框架。

选了后者。原因是目标用户（项目助理、行政人员）的真实需求不是"更强大的 AI"，而是"打开就能用、输出格式固定"。技术上不复杂，但恰好是最匹配场景的方案。产品设计里，克制比堆功能更难。

### 2. 为什么同时做浏览器版和命令行版？

这是一个刻意练习——同一个产品设计，用两种完全不同的交互方式实现，验证设计是否足够清晰、能否适配不同场景。

- **浏览器版**：可视化，双击即用，零配置，面向"我需要马上处理一份文档"
- **命令行版**：管道输入，支持脚本集成，面向"我要批量处理或接入自动化工作流"

### 3. 每种模式定义了"三件套"

每种处理模式都定义了三个东西：输入规范、输出框架、约束规则。这不是随便写的——模拟的是项目助理收到客户需求后的标准工作流：先提炼核心需求、拆解功能点、识别哪些地方需要跟客户确认、给出推进建议。

**本质上是把人的经验变成了 AI 的指令。** 这是整个产品设计的核心思路。

### 4. 也踩过坑

- **v0.1 只有需求整理一个模式**，因为想"先验证可行性"——结果发现这个逻辑能复用到其他文档类型，才逐步扩展到四个模式
- **命令行版在中文 Windows 上报编码错误**，因为 GBK/UTF-8 混用——在 v1.2 加了自动编码检测
- **最初只想做浏览器版**，后来发现脚本场景也需要覆盖，才加了命令行版本——如果一开始就规划两个版本，代码结构会更好

## 快速开始

### 浏览器版

双击 `index.html`，设置 DeepSeek API Key，选择模式，粘贴文档，点击生成。

### 命令行版

```bash
# 文档摘要
python ai-cli.py summary < 输入文件.txt

# 需求整理
python ai-cli.py requirements 需求描述.txt

# 会议纪要
python ai-cli.py minutes 会议记录.txt
```

---

## 四种处理模式

| 模式 | 输入 | 输出 |
|------|------|------|
| 文档摘要 | 长篇文章 | 一句话概括 + 核心要点 + 关键数据 |
| 需求整理 | 客户原始需求 | 核心需求 + 功能点拆解 + 待确认疑问 + 下一步建议 |
| 会议纪要 | 会议记录 | 概要 + 讨论要点 + 决策事项 + 待办清单 |
| 智能识别 | 任意内容 | AI 自动判断类型，选择最合适的整理方式 |

## 技术栈

| 组件 | 方案 | 选择原因 |
|------|------|---------|
| AI 模型 | DeepSeek Chat API | 国内直连，中文理解好 |
| 浏览器端 | 纯 HTML/CSS/JS | 零依赖、零安装、离线可用 |
| 命令行 | Python 标准库 | 无需安装第三方包 |
| 隐私 | API Key 仅存本地浏览器 | 不上传任何服务器 |

## 关于"为什么没有做更复杂的功能"

后续计划里有批量处理、Dify 工作流接入、移动端适配等——但做加法容易，做减法需要理由。

这个项目的意义不在于功能多，而在于它证明：**一个计算机网络专业的学生，可以独立走完 AI 产品从0到1的全流程。** 用户调研做得很浅，原型只是 ASCII 线框，迭代节奏全靠自己判断——都不完美，但整个过程是真实的。

不适合当商业产品，适合当产品能力的样本。
